# BoostaTask

## Overview
Build a minimal RAG (Retrieval-Augmented Generation) API that allows users to upload documents and ask questions over them. The system should retrieve relevant chunks from the document and use an LLM to generate a grounded answer with source references.

## Structure

Почитав про SOLID + в завдані сказано ```Each component has a clear, single responsibility``` тому і вирішив на основі цих правл побудувати структуру

```
Directory structure:
└── BoostaTask/
    ├── main.py - це головний код, звідси все починається
    ├── Dockerfile
    ├── README.md
    ├── compose.yaml
    ├── requirements.txt
    ├── config.py - тут конфіг де можна все налаштовувати, поміняти моделльку або змінити розмір чанку наприклад
    ├── document/ - тут всі функції зв'язані з доукметами або файлами
    │   ├── chunking.py - тут документ розбивається на чанки
    │   ├── main.py - ту головний клас який керує всіма функціями
    │   ├── metadata_storage.py - тут зберігається метадата докумета в бд
    │   ├── parsing.py - тут документ парситься за правилами SOLID тож легко додавати нові формати
    │   └── temporary.py - тут створюється тимчасовий файл
    ├── llm/ - тут ллм
    │   ├── base.py - це базовий клас
    │   └── langchain.py - тут лангчейн з openai, gemini та claude
    ├── routers/ - тут зберігаються всі роутери для фаст апі
    │   ├── chat.py - всі чат ендпойнти
    │   └── document.py - видалення та додавання документу ендройнти
    ├── sql_db/ - тут sql бд
    │   ├── base.py - це базовий клас
    │   └── sqlite.py
    └── vector_db/
        ├── base.py - це базовий клас
        ├── chroma_local.py - векторна бд від chroma на локалці
        └── chroma_web.py - векторна бд від chroma в вебі або в докері (Я її використав)
```


## Pipeline

Кожний ендпойнт захищений x-api-key який можна налаштувати в config.FASTAPI_API_KEY


**POST /document/upload**

Додавання документу

1. Перевірка фоормату файлу - доступні: pdf, docx, txt, xlsx
2. Створення hash_id - тобто перетворення байтів файлу на hash об'єкт sha256, хеш різних файлів різний тому ідеально для перевірки на інування файлу
3. Переврка в бд metadata на існування за hash_id -> якщо вже є повертаєм інформацію про цей файл
4. Створюється Тимчасовий файл який зберігає отриманий файл - використав контекстний менеджер створив клас для контекстного менеджера, причина - навіть якщо щось станеться я можу бути впевненим що файл видалиться також врази зручніше працювати так як не треба відкривати та закривати сессію
5. Парсинг данних відбувається за допомогою бібліотек python-docx, pypdf, pandas але архутектура файлу SOLID тож додавати нові формати дуже зручно
6. Розбиття на чанки. Додаю chunk_overlap, щоб можна було зрозуміти контекст. Використав генератор - так як файли можуть бути великі, щоб не перенавантажувати пам'ять.
7. Додавання до векторної бд. Ембедінг робиться на стороні chromadb але додав варіанти ембедінгу за допомогою ollama та openai. Спочатку думав використати mongodb так як я знаю її, вона дуже добре захищена, плюс в них легко можна розмістити її на сервер, але проблеми почались коли переходив на локалку - тому врішив зупинитись на chromadb швидкий, перевірений, не такий захищений; але дуже зручний. Потім також додав його до докеру тому що тільки веб версія дозволяє асинхроний формат.
8. Зберігання метаданних


**DELETE /document/remove**

Видалення документу

1. Перевірка на уснування в бд
2. Видалення з векторнох бд
3. Видалення з метаданних 



**POST /chat/ask**

Створення запитання

Формат body
{
  "text": "Hello",
  "top_k": 6
}

1. Пошук по векторній бд - ембедінг на стороні chromadb.
2. Форматування чанків в єдиний json формат
3. Запит до ллм - я вибрав фреймворк LangChain так як він дозволяє легко перемикатись між різними ШІ але SOLID тому зручно додавати інші фреймворки


## Launch

Для того щоб запустити потрібно відредагувати config.py

```
llm_service = OpenAILangChainLLMService(
    "Ось тут поміняй на свій апі кей"
)
```
 

```
document_service = document.main.DocumentService(
    parsing.ParserService(),
    chunking.ChunckingService(chunk_size=100, chunk_overlap=10), - Ось тут можна міняти розмір чанку трішки захаркоджено але всеодно легко тому плюс до Chunking strategy is configurable 
    ChromaWebVectorDB(host="chroma", port=8000, collection_name="documents"),
    metadata_storage.MetaDataDB(sql_db)
)
```

Заранити
```
docker compose up
```
